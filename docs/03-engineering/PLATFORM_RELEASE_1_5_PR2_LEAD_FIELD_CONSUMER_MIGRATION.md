## Purpose

This is the pre-schema-change inspection required by ADR-0021. Repository search covered the Prisma schema and migrations, public intake, authenticated lead routes, canonical workspace, legacy admin detail and list routes, portal and portal-token flows, eligibility, jurisdiction, quote/recommendation, application-pack/print and submission routes, uploaded-document consumers, notifications and email/SMS, reports/dashboard metrics, CRM exports, AI adapters, scripts, seed data, fixtures, direct Prisma queries, unit tests, and PostgreSQL integration tests.

Unknown means `null`; it never means false, declined, not applicable, consented, complete, eligible, or ineligible. Public homeowner intake remains governed by its strict path-specific Zod schema even where persistence becomes nullable.

## Consumer Inventory And Compatibility Decision

- Public create: `app/api/intake/route.ts`, `components/LeadForm.tsx`, `lib/validation.ts`, eligibility/AI/quote/pricing, upload, portal token, notifications and audit. The strict public schema remains the authority and writes `HOMEOWNER_INTAKE` plus duplicate normalisation fields.
- Authenticated reads and writes: installer dashboard/list, canonical workspace, legacy admin detail routes/actions, workflow stage service, lead access helpers, dashboard metrics and support views. Nullable facts are formatted as unknown and qualification-dependent conclusions use the completeness service.
- Customer surfaces: portal page/document routes, portal-fill preview and submission-package routes. Existing homeowner records remain unchanged; manual records have no token and action gates block incomplete inputs.
- Documents/exports: application pack and print views, submission package, CRM/raw structured export and uploaded-document readers. Missing facts render as not recorded and governed output fails closed.
- External effects: email, SMS and intake notification orchestration remain invoked only by the existing public intake after its strict validation. Manual creation triggers none.
- Operational data: seed, pilot rehearsal, migration fixtures and all direct Prisma test factories add explicit homeowner origin where they model a homeowner submission; genuinely synthetic generic rows may use `LEGACY_UNKNOWN` only in migration fixtures, never application writes.

## Legacy Assignment Retirement Plan

`assignedAdmin` and `assignedInstaller` remain because the legacy admin detail form still reads and writes them. PR 2 does not dual-write those strings. Canonical manual assignment reads `assignedMembershipId` first and displays the related user name. Retirement requires: (1) every assignment mutation to use the protected membership service, (2) every list/detail/export reader to use membership assignment, (3) a reviewed decision about any ambiguous historical strings, and (4) a later additive-then-removal migration. That cleanup is not part of PR 2.

## Follow-Up Projection Contract

PR 2 writes no work item. PR 3's deterministic backfill and PR 4's service must use the accepted ADR-0020 marker/uniqueness rule so one non-null manual `nextFollowUpAt` produces no more than one open follow-up work item. Compatibility fields remain readable during rollback and are not removed here.

## Privacy And Production Gate

Implementation and disposable non-Production verification are allowed. The platform gate requires recognised `APP_ENV` plus exact `MANUAL_LEAD_CREATION_ENABLED=true` in every environment. Production and Preview therefore default closed, while Development/test require explicit opt-in. Production remains operationally blocked because no authoritative Project Shield and company/privacy-owner record approves the ADR-0021 collection wording, purpose/lawful basis, retention/deletion, follow-up contact, access/correction, sensitive-note, and pilot-minimisation requirements.
