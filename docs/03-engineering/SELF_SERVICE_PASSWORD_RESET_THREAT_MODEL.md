## Scope And Trust Boundaries

This threat model covers the public request form, shared rate-limit store, identity resolution, reset persistence, transactional-email adapter/provider, mailbox link, browser exchange, password-completion transaction, session store, audit, logs, and environment configuration.

Assets are installer account control, credentials, bearer tokens, sessions, tenant data, availability, delivery reputation, audit integrity, personal data, and environment separation. The internet requester, mailbox, email provider, browser, Vercel edge/runtime, PostgreSQL, shared rate-limit store, and Clada operator boundary are separately trusted components.

## Threat Register

| Asset | Threat | Attack path | Impact | Mitigation | Residual risk | Validation evidence required |
| --- | --- | --- | --- | --- | --- | --- |
| Account existence | Enumeration | Compare status, body, redirect, timing, or email-related UI for known and unknown addresses. | Targeted phishing and credential attacks. | Same neutral `202`, copy, redirect, and materially similar bounded path; no public account audit/status. | Provider-side timing and actual email receipt remain observable to the mailbox owner. | Automated known/unknown/inactive comparisons across repeated samples; response snapshot. |
| Reset bearer | Token guessing | Brute force exchange/completion endpoints. | Account takeover. | 256-bit CSPRNG token, 30-minute expiry, keyed digest only, rate limits, generic rejection. | Compromise is possible if token is stolen rather than guessed. | Entropy/length unit tests; invalid-token rate tests; no raw database value. |
| Reset bearer | Token theft | Mailbox compromise, forwarded email, malicious extension, local device compromise. | Account takeover. | Short expiry, newest-only, one-time use, clean URL, scoped `HttpOnly` cookie, security notice, all-session revocation. | A compromised mailbox/device within 30 minutes can complete reset. | Browser exchange test; cookie inspection; email-content review; residual risk accepted. |
| Reset bearer | Replay | Reuse a consumed token or race two completions. | Multiple resets or takeover. | Conditional atomic consumption inside serializable password/session transaction. | Database availability errors may deny both attempts, safely. | Concurrent PostgreSQL integration test proves exactly one commit. |
| Reset bearer | Leakage through logs/errors/analytics | Query/path logging, request-body dumping, exception capture, audit metadata, client telemetry. | Account takeover during lifetime; privacy incident. | Fragment link, bounded POST, no body logging, sensitive-key sanitiser, no reset telemetry/third parties, structured safe errors. | Browser extensions and provider click scanners are outside full control. | Source scan; runtime log scan; audit/database inspection; error-injection tests. |
| Reset bearer | Referrer leakage | Reset page navigates or loads third-party resource while token is present. | Token disclosure to another origin. | Fragment exchange and immediate history cleanup; `Referrer-Policy: no-referrer`; no third-party resources; restrictive CSP. | User may manually copy the original email URL. | Header assertions and browser network trace showing no third-party request/referrer. |
| Link destination | Malicious Host/header construction | Attacker supplies `Host` or forwarded-host so email points to attacker. | Token theft and takeover. | Exact environment-owned canonical origin; ignore request host and forwarded headers. | Misconfigured canonical origin can still misroute links. | Unit/integration tests inject host variants; deployment configuration review. |
| Delivery channel | Email flooding | Repeated requests to one account or many addresses. | User harassment, provider reputation damage, cost. | Per-IP, keyed-email, and account atomic limits; newest-only request; neutral limited result. | Distributed botnets can spread IPs. | Cross-instance-store integration tests and controlled threshold tests. |
| Database availability | Database flooding | Submit unbounded unique addresses or create many rows. | Storage/connection exhaustion. | No row for unresolved address; one active row per user; account/IP limits; 30-day cleanup; bounded inputs. | Many eligible compromised accounts can create retained rows. | Volume test, query-plan/index review, cleanup test, row-growth calculation. |
| Password/session integrity | Race condition | Request while completing, complete while superseding, or two password submissions. | Stale token succeeds or sessions survive. | Transactional supersession and conditional consumption; completion revalidates eligibility and active state; all session deletes in transaction. | Serializable retries may affect availability. | Concurrency matrix in PostgreSQL integration tests. |
| Active sessions | Session persistence after reset | Password changes but old `AuthSession` rows remain. | Attacker retains access. | Password update, token consumption, all-session deletion, and audit in one transaction; rollback on failure. | An already executing request may finish before revocation commits. | Session-row and live-cookie tests before/after reset; injected delete failure rollback. |
| Tenant data | Cross-tenant access | Supply organisation ID, exploit ambiguous membership, or keep a session from another organisation. | Tenant data breach. | No client organisation input; server resolves current unique active membership/organisation/Installer; all user sessions revoked. | Future multi-membership model requires redesign. | Two-tenant integration tests; schema uniqueness check; direct-ID denial tests. |
| Inactive identity | Inactive-account recovery | Reset inactive user, membership, or organisation. | Bypass suspension/offboarding. | Eligibility checked on request, exchange, and completion; fail closed. | Status can change after email delivery, causing safe denial. | State-transition integration tests. |
| Mailbox/account | Compromised mailbox | Attacker with email access resets password. | Account takeover. | Security notice, short expiry, one use, normal login required, audit, session revocation; exceptional support response. | Email possession remains the recovery proof for pilot. | Product/security approval and incident runbook review. |
| Credential | Weak-password selection | User chooses common, contextual, blank, oversized, or current password. | Credential compromise or resource abuse. | Existing server policy, 12–128 bounds, confirmation, current-hash comparison, Argon2id. | Offline breached-password screening is not yet implemented. | Unit/integration tests for every policy class and hash verification. |
| Service availability | Denial of service | Expensive account lookup, hashing, rate-store failure, provider latency, oversized body. | Recovery/login degradation and cost. | Request-size limits, cheap rate limit before account work, bounded provider timeout, database indexes, no password hash on request, fail-closed issuance. | Shared store/provider outage blocks recovery but not normal login. | Load/boundary tests and timeout/failure injection. |
| Audit/privacy | Audit-data leakage | Store email, IP, token/digest, password/hash, cookie, headers, or provider payload. | Security and GDPR incident. | Allowlisted metadata schema plus existing sanitiser; IDs and safe reason codes only; 30-day reset-record retention. | User IDs can still be personal data and require controlled access. | Audit snapshot/redaction tests; privacy review; retention purge evidence. |
| Environment isolation | Preview-to-Production confusion | Preview code emits Production link, sends to Production user, or shares provider/rate keys. | Production account takeover, privacy breach, unsafe testing. | Exact environment-origin validation, separate credentials/stores, Preview sandbox and recipient allowlist, Production hostname forbidden outside Production. | Operator configuration error remains possible before fail-closed validation. | Configuration unit tests and Preview deployment acceptance with synthetic recipients. |
| Delivery state | Provider succeeds but persistence fails | Provider accepts mail but request remains pending; link arrives unusable. | Confusing user experience, support load. | A request is valid only after durable dispatch state; neutral retry; safe correlation evidence. | One unusable email can be delivered. | Failure injection after provider acceptance; subsequent request succeeds and supersedes. |
| Delivery state | Persistence succeeds but provider fails | Token exists before failed email. | Hidden usable token or future accidental delivery. | Pending requests are unusable; mark failed/revoked; raw token discarded; newest request supersedes. | Failure to persist revocation leaves pending but unusable record. | Provider timeout/error integration tests and database state assertions. |
| Proxy identity | Spoofed forwarded IP | Client supplies arbitrary `X-Forwarded-For` to evade limits or target another IP. | Flooding or denial of legitimate users. | Accept only the documented Vercel-trusted client IP signal; ignore arbitrary chains; account/email limits remain. | Provider trust-chain changes require review. | Header-spoof tests in Preview and documented deployment evidence. |
| Browser cache/history | Secret persistence | Browser history, back-forward cache, shared device, cached HTML. | Token/password exposure. | Fragment removal before form, `no-store`, no password persistence, reset cookie cleared on terminal states. | Email history still contains original link. | Browser history/cache inspection and response-header tests. |

## Abuse And Privacy Logging

Resolved-account security events use user and organisation IDs. Unknown-address traffic may generate aggregate operational metrics keyed by short-lived purpose-specific digests, but never a durable per-email audit record. Raw IP addresses are not stored in `AuditLog`; the rate store retains pseudonymous keys only for their TTL. Alert thresholds must be high enough to avoid turning normal mistakes into privacy-heavy incident records.

Rejected-token audit is rate-bounded. It records a reset-record ID only when a safe digest lookup resolves one; wholly random tokens produce aggregate rejection metrics rather than unbounded audit rows.

## Security Acceptance Gate

The feature cannot ship until every mitigation above has implementation evidence, the open provider/sender/rate-store approvals in ADR-0023 are resolved, and there is no unresolved Critical or pilot-blocking High finding. Any exception requires a documented CTO security decision; neutral responses, raw-token non-persistence, atomic session revocation, canonical origins, and tenant isolation are not waivable pilot controls.
