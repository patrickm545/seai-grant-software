Use this checklist before a platform release is approved for merge.

CEO approval confirms that the release is worth shipping from a business, customer, commercial, scope, risk, and timing perspective. It does not replace CTO engineering review.

## Customer Value

1. Does the release improve a real customer, operator, installer, or platform workflow?
2. Is the value clear enough to explain without implementation details?
3. Does the release help Clada Systems serve contractor-led businesses?
4. Is the SolarGRANT Pro proving slice, if any, valuable and appropriately narrow?

## Business Alignment

1. Does the release support the Clada Systems mission?
2. Does it strengthen Clada OS as a reusable platform?
3. Does it preserve SolarGRANT Pro as the first product module rather than the whole platform?
4. Does it fit the approved platform roadmap or an approved roadmap change?

## Scope

1. Is the release scope clear?
2. Are non-goals clear?
3. Has scope creep been avoided?
4. Are deferrals acceptable?
5. Is anything important missing from the Master Release Specification?

## Cost

1. Is the engineering cost justified by customer value, risk reduction, or strategic platform value?
2. Does the release avoid overbuilding?
3. Are operational and maintenance costs understood?
4. Is any new dependency justified?

## Timeline

1. Is this the right release to do now?
2. Does the timeline support SolarGRANT Pro and Clada OS priorities?
3. Are review, validation, and release steps realistic?
4. Does this release block or unblock another important milestone?

## Commercial Value

1. Does the release improve readiness for pilots, sales, onboarding, retention, operational delivery, or future product expansion?
2. Does it reduce risk in a way that protects commercial trust?
3. Does it create a reusable foundation for future revenue-generating work?

## Risk

1. Are customer, privacy, security, compliance, operational, and reputation risks understood?
2. Are known risks acceptable?
3. Are rollback or mitigation options understood?
4. Are deferred risks acceptable for this release?

## Dependencies

1. Does the release depend on unfinished lower-level platform capabilities?
2. Are product, provider, migration, data, team, or review dependencies clear?
3. Are dependency risks acceptable?

## Release Readiness

1. Has CTO review completed?
2. Has validation completed or been explicitly caveated?
3. Is the draft PR clear enough to review?
4. Are known risks and deferrals documented?
5. Is the release ready to merge without hidden follow-up work?
